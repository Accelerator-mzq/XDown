# 问题定位与修复（含模型升级流程）

> **路径规则**：所有文档路径以 `smartroute.config.json` 中的 `documents` 配置为准，不要使用硬编码路径。

用户报告了一个问题：$ARGUMENTS

## 第一步：读取配置
1. 读取 `smartroute.config.json` 中 `documents` 配置，确定以下路径：
   - `system_test_cases`：系统测试用例文档路径
   - `unit_test_cases`：单元测试用例文档路径

## 第二阶段：定位（当前模型）
1. 分析问题描述，复现问题
2. 查看相关代码和日志，定位根因
3. 如果能定位到根因，跳到第三阶段
4. 如果经过多轮尝试仍无法定位，明确输出：
   "⚠️ 当前模型定位困难，建议切换到 opus 模型继续分析"
   并附上已排查的信息和思路

## 第三阶段：修复
1. 根据根因给出修改方案
2. 执行代码修改
3. 确保编译通过
4. 运行相关测试确认修复

## 第四阶段：漏测分析 — 双轨覆盖率审查（需要 opus 模型）
问题修复后，必须执行以下两层审查，并在回复中分别汇报：

### A. 系统级覆盖审查
1. 读取 `documents.system_test_cases` 指定的文档
2. 检查该 Bug 的用户场景是否在系统测试用例中
3. 如果属于漏测缺陷：主动修改文档，补充新的 ST 用例
4. 如果已存在：分析为什么会漏测，将原因记录在对应测试例的备注中

### B. 单元级覆盖审查
1. 读取 `documents.unit_test_cases` 指定的文档
2. 查阅 `tests/` 目录下相关的测试文件（例如报了数据库 Bug 就查 `tst_dbmanager.cpp`）
3. 分析：为什么之前的单元测试没有拦截到这个底层的逻辑/边界错误？
4. 如果缺失了特定的断言 (Assert) 或边界条件 Mock，必须直接在对应的 C++ 测试文件中补齐
5. 保存更新后的测试例文档

## 第五阶段：强制自动化测试实现
1. 优先在 `tests/` 目录下编写对应的 C++ QtTest 单元测试，或补充到 `Run-Test.ps1` 无头命令行测试模式中
2. 编写完成后，必须自己调用 `Run-Test.ps1` 或 `ctest` 跑通该测试，证明 Bug 已被修复
3. **豁免条款**：如果该 Bug 属于纯粹的视觉 UI 问题（如颜色不对、弹窗位置偏移、QML 渲染错位），无法通过无头模式和 C++ 断言完成自动化测试，必须在回复中明确说明："**无法完成自动化测试，原因：[具体的技术限制]**"，并提供详细的手动验证步骤

## 第六阶段：回归测试（切回 minimax）
1. 遍历所有系统测试例，执行自动化测试
2. 运行所有单元测试
3. 确保无新增失败
4. 输出回归测试报告到 .pipeline/regression-report.md
