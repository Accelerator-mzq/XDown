## 1. 产品概述 (Product Overview)

- **产品定位**：XDown 是一款主打极简、无广告、现代化的本地纯粹下载工具，专注于提供高性能、多协议支持的下载体验。

- **本期目标 (v2.0)**：在 v1.0 单线程 HTTP/HTTPS 下载的基础上，实现**多线程分块下载**以大幅提升下载速度，并引入 **P2P 协议支持**（BT 种子、磁力链接），同时优化磁盘管理和文件完整性校验机制。

- **核心价值主张**：
  - **性能飞跃**：通过多线程并发下载，解决"下得慢"的痛点。
  - **协议全面**：支持 HTTP/HTTPS、BitTorrent、Magnet 等主流下载协议。
  - **极客友好**：为高级用户提供 BT/Magnet 专属管理界面和完整性校验功能。

- **目标用户**：对下载速度和协议支持有较高要求的桌面用户，包括需要下载大型文件、Linux 镜像、开源软件的技术用户。

## 3. 核心功能需求 (Functional Requirements)

### 3.1 任务创建模块 (Task Creation)

#### 3.1.1 多协议 URL 解析

- **HTTP/HTTPS 协议**：支持标准 `http://` 和 `https://` 协议的链接解析（继承自 v1.0）。
- **Magnet 磁力链接**：支持解析 `magnet:?` 格式的磁力链接，提取 `xt`（文件哈希）、`dn`（显示名称）、`tr`（Tracker 地址）等参数。
- **Torrent 种子文件**：支持通过文件选择器导入本地 `.torrent` 文件，解析种子元数据（文件列表、Tracker、创建者信息）。

#### 3.1.2 路径选择与文件名解析

- **默认保存路径**：系统提供默认保存路径（如 Windows 的 `~/Downloads`），用户可通过系统原生文件管理器更改路径。
- **文件名自动解析**：
  - HTTP/HTTPS：从 URL 或 `Content-Disposition` 响应头中提取文件名。
  - Magnet：从 `dn` 参数提取文件名，若无则使用哈希值作为临时名称。
  - Torrent：从种子文件的 `info.name` 字段提取文件名。

#### 3.1.3 重复性校验

- 新建任务时，需校验当前下载列表中是否存在相同 URL/Magnet/InfoHash 且处于"下载中"或"等待中"的任务，若有则阻断创建并提示用户。

#### 3.1.4 高级选项（新增）

- **线程数设置**：用户可自定义 HTTP/HTTPS 下载的线程数（默认 8 线程，范围 1-32）。支持"自动"模式，系统根据网络状况和服务器性能自适应调整线程数。
- **分块大小设置**：用户可自定义每个分块的大小（默认 1MB，范围 512KB-10MB）。
- **完整性校验选项**：用户可选择"自动校验"（默认）或"跳过校验"。跳过校验可加快下载完成速度，但可能导致文件损坏风险。
- **BT 做种策略**：用户可选择"下载完成后自动做种"（默认）或"下载完成后自动停止做种"。

### 3.2 核心下载引擎 (Download Engine)

#### 3.2.1 多线程分块下载 (HTTP/HTTPS)

**目标**：通过并发下载多个文件分块，大幅提升下载速度。

**实现机制**：

1. **服务器能力检测**：
   - 发送 HEAD 请求，检查服务器是否支持 `Accept-Ranges: bytes`。
   - 若支持，获取文件总大小（`Content-Length`）。
   - 若不支持，回退到单线程下载模式。

2. **动态分块策略**：
   - 根据文件大小和用户设置的线程数，将文件切分为多个 Chunk。
   - 每个 Chunk 的大小 = 文件总大小 / 线程数（最后一个 Chunk 可能略大）。
   - 示例：4.7 GB 文件，8 线程，每个 Chunk 约 587.5 MB。

3. **并发下载**：
   - 为每个 Chunk 创建独立的 HTTP 连接，使用 `Range: bytes=start-end` 请求头指定下载范围。
   - 每个线程将下载的数据写入临时文件（如 `filename.part1`, `filename.part2`）。

4. **进度聚合**：
   - 主线程定时（每秒）汇总所有 Chunk 的下载进度，计算全局进度和速度。
   - 更新 UI 显示的进度条和速度信息。

5. **文件合并**：
   - 所有 Chunk 下载完成后，按顺序将临时文件合并为最终文件。
   - 合并完成后删除所有临时文件。

6. **断点续传**：
   - 若下载中途暂停或中断，保存每个 Chunk 的已下载字节数到数据库。
   - 恢复下载时，每个线程从上次断开的位置继续下载。

**异常处理**：

- 若某个 Chunk 下载失败（网络错误、超时），自动重试 3 次。
- 若重试失败，暂停整个任务并标记为"错误"状态。

**自适应线程数调整机制**（新增）：

1. **初始线程数**：默认使用 8 线程启动下载。
2. **性能监控**：每 10 秒采样一次下载速度，计算平均速度和速度增长率。
3. **动态调整策略**：
   - 若速度增长率 < 10%，且当前线程数 < 32，则增加 2 个线程。
   - 若速度增长率 < 0（速度下降），且当前线程数 > 4，则减少 2 个线程。
   - 若速度稳定（增长率在 -5% ~ +10% 之间），保持当前线程数。
4. **调整上限**：单个任务最多 32 线程，最少 1 线程。
5. **用户控制**：用户可在设置中关闭自适应调整，手动固定线程数。

#### 3.2.2 P2P 下载协议支持 (BitTorrent & Magnet)

**目标**：支持 BT 种子和磁力链接下载，实现 P2P 文件共享。

**实现机制**：

1. **第三方库集成**：
   - 引入 `libtorrent` 库（推荐版本：libtorrent-rasterbar 2.0+）。
   - 通过 CMake 配置 `libtorrent` 的静态链接或动态链接。

2. **种子文件解析**：
   - 使用 `libtorrent::torrent_info` 解析 `.torrent` 文件。
   - 提取文件列表、Tracker 地址、创建者信息、文件哈希（InfoHash）。

3. **磁力链接解析**：
   - 使用 `libtorrent::parse_magnet_uri` 解析 `magnet:?` 链接。
   - 提取 InfoHash、显示名称、Tracker 地址。

4. **Tracker 与 DHT 支持**：
   - 连接 Tracker 服务器获取 Peer 列表。
   - 启用 DHT（分布式哈希表）以支持无 Tracker 的磁力链接下载。

5. **Peer 连接与数据传输**：
   - 与多个 Peer 建立连接，交换文件分块（Piece）。
   - 实现 BitTorrent 协议的 Choke/Unchoke、Request/Piece 消息处理。

6. **上传与做种**：
   - 下载完成后，根据用户设置决定是否自动切换为做种模式（Seeding）。
   - 若用户选择"下载完成后自动做种"（默认），任务自动切换为做种模式，向其他 Peer 上传文件分块。
   - 若用户选择"下载完成后自动停止做种"，任务下载完成后直接移至"已完成"列表，不进行上传。
   - 用户可在设置中配置上传速度限制、做种时间限制（如做种 24 小时后自动停止）、分享率限制（如分享率达到 2.0 后自动停止）。

7. **进度与速度显示**：
   - 实时显示下载速度、上传速度、已连接 Peer 数量、可用 Seed 数量。
   - 显示文件完整性进度（已下载的 Piece 数量 / 总 Piece 数量）。

**异常处理**：

- 若无法连接到任何 Peer，提示"无可用 Peer，请检查网络或 Tracker 状态"。
- 若磁力链接无法解析元数据，提示"磁力链接无效或元数据获取失败"。

#### 3.2.3 磁盘与文件管理优化

**目标**：优化磁盘 I/O 性能，防止磁盘碎片化，并确保文件完整性。

**实现机制**：

1. **磁盘预分配**：
   - 下载启动前，根据文件总大小，使用 `fallocate`（Linux）或 `SetFileValidData`（Windows）预先向操作系统申请全量磁盘空间。
   - 优点：防止磁盘碎片化，提升写入性能。
   - 注意：需检查目标磁盘剩余空间是否充足（保留 100MB 缓冲空间）。

2. **完整性校验**：
   - **HTTP/HTTPS 下载**：
     - 若服务器提供 `ETag` 响应头，下载完成后重新请求 HEAD 并对比 ETag。
     - 若服务器提供 `Content-MD5` 响应头，计算本地文件的 MD5 并对比。
     - 用户可在设置中选择"跳过完整性校验"以加快下载完成速度（不推荐，可能导致文件损坏）。
   - **BT 下载**：
     - 使用 `libtorrent` 内置的 Piece Hash 校验机制，确保每个 Piece 的 SHA-1 哈希值正确。
     - 下载完成后，自动校验所有 Piece，若有损坏则重新下载。
     - BT 下载的完整性校验无法跳过（协议强制要求）。

3. **临时文件管理**：
   - 多线程下载的临时文件统一存放在 `<保存路径>/.xdown_temp/` 目录下。
   - 下载完成后自动清理临时文件。
   - 若软件异常退出，下次启动时自动检测并清理孤立的临时文件。

### 3.3 任务控制与状态流转 (Task Control)

#### 3.3.1 状态定义

任务在其生命周期内包含以下状态：

- **等待中 (Waiting)**：任务已创建，但因并发限制尚未开始下载。
- **下载中 (Downloading)**：任务正在下载，网络连接活跃。
- **已暂停 (Paused)**：用户主动暂停，或因错误自动暂停。
- **已完成 (Completed)**：文件下载完成，已通过完整性校验。
- **做种中 (Seeding)**：仅 BT 任务，下载完成后继续上传。
- **错误 (Error)**：下载过程中发生不可恢复的错误。

#### 3.3.2 并发控制

- **HTTP/HTTPS 任务**：最多同时进行 5 个下载任务（v1.0 为 3 个，v2.0 提升至 5 个）。
- **BT/Magnet 任务**：最多同时进行 3 个下载任务（P2P 下载占用更多网络资源）。
- 超过限制的任务进入"等待中"队列，按创建时间先后顺序自动启动。

#### 3.3.3 用户指令

- **开始/继续**：触发网络连接，进入"下载中"状态。
- **暂停**：主动断开网络连接，保存已下载的临时文件和进度信息，进入"已暂停"状态。
- **删除**：从任务列表中移除记录。需弹窗二次确认："是否同时删除本地文件？"
- **打开文件夹**：仅已完成任务可用，打开文件所在目录并高亮文件。
- **停止做种**：仅 BT 任务可用，停止上传并将任务移至"已完成"列表。

#### 3.3.4 批量操作（新增）

- **Start All**：一键启动所有"已暂停"和"等待中"的任务（受并发限制约束）。
- **Pause All**：一键暂停所有"下载中"的任务。

### 3.4 任务列表展示 (List Display)

#### 3.4.1 数据绑定

- 通过 `QAbstractListModel` 实现底层数据到 QML 界面的映射（继承自 v1.0）。
- 新增 BT 任务专属字段：上传速度、Peer 数量、Seed 数量、做种时间。

#### 3.4.2 卡片元素

每个任务卡片必须实时展示以下信息：

- **文件名** (File Name)
- **当前下载速度** (如 12 MB/s)
- **当前上传速度** (仅 BT 任务，如 500 KB/s)
- **文件大小与进度** (如 2.1 GB / 4.7 GB)
- **进度条视觉化展现** (0% - 100%)
- **预计剩余时间 (ETA)** (如 ETA: 03:20)
- **状态标签** (Downloading、Completed、Paused、Seeding、Error)
- **操作按钮** (根据状态显示：暂停、开始、删除、打开文件夹、停止做种)

#### 3.4.3 分类筛选

- **Downloading**：显示所有"下载中"和"等待中"的任务。
- **Completed**：显示所有"已完成"的任务（不包括做种中的 BT 任务）。
- **BT / Magnet**：显示所有 BT 和 Magnet 任务（包括下载中、做种中、已完成）。
- **Trash**：显示所有已删除但未清空的任务（回收站功能）。
  - 用户可手动恢复任务或永久删除。
  - 自动清空机制：回收站中的任务在 30 天后自动永久删除（可在设置中配置清空周期：7天/30天/永不清空）。
  - 永久删除时，若用户选择"同时删除本地文件"，则本地文件也会被删除。

---

### 3.5 数据持久化 (Data Persistence)

#### 3.5.1 实时更新

- 当任务状态发生改变（新建、暂停、完成、错误）时，必须将状态写入 SQLite 数据库。
- 新增字段：线程数、分块大小、上传速度、Peer 数量、Seed 数量、做种时间。

#### 3.5.2 进度节流

- 下载过程中的进度更新每 2 秒批量写入数据库一次，避免频繁写盘（继承自 v1.0）。
- 速度信息实时更新（仅内存，不写入数据库）。

#### 3.5.3 重启恢复

- 软件启动时，从 SQLite 读取所有历史任务。
- 未完成的任务默认置为"已暂停"状态，用户可手动恢复下载。
- BT 任务恢复时，自动重新连接 Tracker 和 DHT，恢复 Peer 连接。

## 4. 性能与非功能需求 (Non-functional Requirements)

### 4.1 线程隔离与并发性能

1. **线程隔离**：界面渲染（Main Thread）与文件下载、数据库读写（Worker Thread）必须严格物理隔离。UI 绝对不能因为大文件写入或多线程下载而发生卡顿或无响应。

2. **多线程下载性能**：
   - 8 线程下载时，总下载速度应达到单线程的 5-8 倍（取决于网络带宽和服务器性能）。
   - 每个线程的 CPU 占用率应控制在 5% 以内（8 线程总计不超过 40%）。

3. **P2P 下载性能**：
   - BT 下载时，应能同时连接至少 50 个 Peer。
   - DHT 节点发现延迟应控制在 10 秒以内。

### 4.2 资源占用

1. **内存占用**：
   - v2.0 单个 HTTP/HTTPS 任务（8 线程）的内存占用峰值不得超过 100MB。
   - 单个 BT 任务的内存占用峰值不得超过 150MB。
   - 软件空闲时（无下载任务）的内存占用应控制在 50MB 以内。

2. **磁盘 I/O**：
   - 多线程下载时，磁盘写入速度应能充分利用 SSD 性能（至少 200 MB/s）。
   - 避免频繁的小块写入，使用缓冲区批量写入（每次至少 64KB）。

3. **网络带宽利用率**：
   - 多线程下载应能充分利用用户的网络带宽（至少达到 90% 利用率）。
   - 支持用户自定义全局下载速度限制和上传速度限制。

### 4.3 界面刷新率

- 前端界面的下载速度和进度条刷新频率应控制在 **1次/秒**，过高会导致 CPU 占用飙升且无实际意义（继承自 v1.0）。
- 全局速度监控（顶部右上角）的刷新频率同样为 **1次/秒**。

### 4.4 稳定性与容错

1. **网络异常处理**：
   - 支持自动重连机制，网络中断后自动重试（最多 3 次，间隔 5 秒）。
   - 若服务器返回 5xx 错误，自动暂停任务并提示用户。

2. **磁盘空间不足**：
   - 下载前检查目标磁盘剩余空间，若不足则阻止任务启动并提示用户。
   - 下载过程中若磁盘空间不足，立即暂停任务并提示用户清理磁盘。

3. **崩溃恢复**：
   - 软件异常退出后，下次启动时自动恢复所有未完成的任务到"已暂停"状态。
   - 临时文件和进度信息应能完整恢复，避免重新下载。

### 4.5 安全性

1. **HTTPS 证书验证**：
   - 默认启用 HTTPS 证书验证，防止中间人攻击。
   - 用户可在设置中选择"忽略证书错误"（仅用于特殊场景，需显著警告）。

2. **文件完整性校验**：
   - 所有下载完成的文件必须通过完整性校验（ETag、MD5、SHA-1）。
   - 若校验失败，提示用户"文件可能已损坏，建议重新下载"。

3. **恶意种子防护**：
   - 解析 `.torrent` 文件时，检查文件大小是否异常（如单个文件超过 1TB）。
   - 若检测到异常，提示用户"种子文件可能存在风险，请谨慎下载"。

  - **libtorrent-rasterbar 2.0+**：用于 BT 和 Magnet 协议支持。
  - **OpenSSL 1.1.1+**：用于 HTTPS 加密通信（Qt 依赖）。

- **支持的编译器 (Windows 主力开发平台)**：
  - **首选**：MSVC 2022 (v143工具集) 64-bit。
  - **备选**：MinGW-w64 (GCC 11.2 或以上) 64-bit。
  - _(注：严禁使用 32 位编译器，以突破单文件 4GB 的内存寻址限制)_。

- **集成开发环境 (IDE)**：Qt Creator 11.0+ 或 Visual Studio 2022（需安装 Qt Visual Studio Tools）。

- **本地数据库引擎**：SQLite 3（直接使用 Qt SQL 模块自带的驱动，无需额外编译）。

### 6.2 运行与测试环境 (Execution & Testing Environment)

此环境定义了 XDown v2.0 承诺能够正常运行的用户电脑下限，也是 QA 测试阶段必须覆盖的基础环境。

- **操作系统要求 (OS)**：
  - **Windows**：Windows 10 (21H2 及以上版本) 或 Windows 11（仅支持 64 位）。
  - _(注：v2.0 暂不将 macOS 和 Linux 列入强制测试环境，待后续迭代处理跨平台问题)_。

- **最低硬件要求 (Minimum Hardware)**：
  - **CPU**：四核处理器，2.0 GHz 及以上（支持 x86-64 指令集）。
  - **内存 (RAM)**：8 GB（多线程下载和 P2P 下载需要更多内存）。
  - **系统盘空间**：200 MB 可用空间（用于安装软件、存放 SQLite 数据库文件和临时文件）。

- **推荐硬件要求 (Recommended Hardware)**：
  - **CPU**：六核或以上处理器，3.0 GHz 及以上。

  - 需要标准 IPv4/IPv6 宽带网络接入。
  - 测试阶段需覆盖：正常局域网、存在代理/梯子的复杂网络环境、以及突发断网情况下的软件健壮性测试。
  - **P2P 下载特殊要求**：需要路由器支持 UPnP 或手动配置端口转发，以确保 BT 下载能正常连接 Peer。

## 7. 技术风险与待确认问题 (Technical Risks & Open Questions)

### 7.1 技术风险

1. **libtorrent 集成复杂度**：
   - libtorrent 是一个庞大的 C++ 库，编译和集成可能遇到依赖问题（如 Boost 库版本冲突）。
   - **缓解措施**：使用预编译的 libtorrent 二进制包，或通过 vcpkg/Conan 等包管理器简化集成。

2. **多线程下载的文件合并性能**：
   - 大文件（10GB+）的多线程下载完成后，合并临时文件可能耗时较长（数十秒）。
   - **缓解措施**：使用内存映射文件（mmap）或异步 I/O 优化合并性能。

3. **P2P 下载的网络穿透问题**：
   - 部分用户的路由器不支持 UPnP，且未配置端口转发，导致无法连接 Peer。
   - **缓解措施**：提供详细的端口转发配置指南，并支持 DHT 和 PEX（Peer Exchange）协议以提高连接成功率。

4. **跨平台兼容性**：
   - v2.0 暂不支持 macOS 和 Linux，但未来需要考虑跨平台移植。
   - **缓解措施**：代码设计时尽量使用 Qt 的跨平台 API，避免使用 Windows 特有的 API。
